# Test From Contract Workflow

Contract YAML から TDD テストコードを生成するワークフロー。
Level 1（構造検証、即 GREEN）と Level 2（実装検証、RED スタブ）の 2 レベル構造。

> **前提**: `contract-schema.md` と `blueprint-structure.md` を参照。

## ワークフロー（6 ステップ）

### Step 1: コンテキスト読み込み

プロジェクトの状態を把握する。

```
実行内容:
1. git root 検出
2. .blueprint/contracts/ スキャン
3. 各 Contract の id, type, status, version を一覧化
4. 依存関係の把握（links.depends_on）
```

**ディレクトリ → タイプマッピング**:

| ディレクトリ | Contract type | 例 |
|------------|---------------|-----|
| `.blueprint/contracts/api/` | api | CON-order-create |
| `.blueprint/contracts/external/` | external | CON-stripe-payment-intent |
| `.blueprint/contracts/files/` | file | CON-product-bulk-import |
| `.blueprint/contracts/internal/` | internal | CON-metrics-collector |

> **注意**: `type: file` のディレクトリ名は `files/`（複数形）。
> **注意**: `internal` は `subtype`（service / repository）で テスト生成パターンが分岐する。

### Step 2: Contract 選択

テスト生成対象を決定する。

```
デフォルト: status が active または draft の全 Contract
選択モード:
- 全件（デフォルト）
- ユーザー指定（Contract ID リスト）

ユーザーに一覧を提示して承認を得る（必須）。
```

**一覧表示フォーマット**:

```
| # | Contract ID                  | Type     | Status | Version | Depends On        |
|---|------------------------------|----------|--------|---------|-------------------|
| 1 | CON-order-create             | api      | draft  | 1.0.0   | CON-stripe-...    |
| 2 | CON-stripe-payment-intent    | external | draft  | 1.0.0   | —                 |
| 3 | CON-product-bulk-import      | file     | draft  | 1.0.0   | —                 |
```

### Step 3: テスト環境確認

テストフレームワークと出力先を決定する。

```
フレームワーク自動検出:
1. vitest.config.* → Vitest (TypeScript/JavaScript)
2. jest.config.* / package.json の jest フィールド → Jest (TypeScript/JavaScript)
3. conftest.py / pyproject.toml [tool.pytest] → pytest (Python)
4. *_test.go → Go testing (Go)
5. 未検出 → ユーザーに確認（デフォルト推奨: Vitest）

出力先: tests/contracts/ (デフォルト、ユーザー変更可)
```

**フレームワーク別の RED スタブ形式**:

| フレームワーク | RED スタブ | 拡張子 |
|-------------|-----------|--------|
| Vitest | `expect.fail("RED: ...")` | `.test.ts` |
| Jest | `expect(() => {}).toThrow(); // RED: ...` または `throw new Error("RED: ...")` | `.test.ts` |
| pytest | `pytest.fail("RED: ...")` | `_test.py` |
| Go testing | `t.Fatal("RED: ...")` | `_test.go` |

### Step 4: Level 1 テスト生成（構造検証、即 GREEN）

Contract YAML の構造を検証するテストを生成する。実装不要で即パスする。

```
生成内容:
1. contract-loader ヘルパー（YAML パースユーティリティ、初回のみ）
2. 各 Contract ごとの構造検証テスト

共通チェック項目:
- YAML がパース可能であること
- 必須メタデータ: id, type, version, status, owner, updated_at
- links 構造: implements, depends_on, decided_by, impacts が配列
- id が CON-* プレフィックスであること

タイプ別チェック項目:
- api: method, path, input, output, business_rules が存在
- external: provider, endpoint, request, response, constraints が存在
- file: direction, format, columns, processing_rules, result が存在
- internal: subtype, description, input, rules が存在
  - subtype: service の場合: state または side_effects のいずれかが存在（オプション）
  - subtype: repository の場合: storage が存在（オプション）

出力: tests/contracts/level1/CON-{name}.test.ts (or 対応拡張子)
```

### Step 5: Level 2 テスト生成（実装検証、RED スタブ）

Contract のフィールドからテストケースを機械的に導出する。
全テストは RED スタブ（実行すると FAIL）として生成する。

**AAA 骨格（必須）**: 全 Level 2 テストに Arrange / Act & Assert のコメント骨格を含める。
GREEN 化する開発者がテストの意図と構造を即座に理解できるようにする。

**矛盾検出**: `required: true` + `default` が同一フィールドに指定されている場合は P1 finding として報告する。

```
テスト導出パターン（contract-schema.md 準拠）:

共通:
- required: true → 空値/欠落テスト
- min/max: N → 境界値テスト (N-1, N, N+1)
- pattern → マッチ/不一致テスト
- enum → 全有効値 + 無効値テスト
- default → 省略時デフォルト適用テスト

api 固有:
- business_rules[] → 各 BR-ID の正常/異常テスト
- state_transition → 初期状態 + 許可遷移 + 拒否遷移テスト
- errors[] → 各エラーの status + code テスト

external 固有:
- constraints[] → 各 EC-ID の正常/異常テスト（冪等性、タイムアウト等）
- response.errors → 各エラー type のハンドリングテスト
- 固定値フィールド（value: "jpy" 等）→ 値の検証テスト

file 固有:
- columns[].max_length → 文字数境界テスト
- max_rows → 行数境界テスト
- has_header → ヘッダー有無テスト
- processing_rules[] → 各 PR-ID の正常/異常テスト
- result 構造 → 正常/エラー結果形式テスト

internal 固有（subtype で分岐）:
  共通:
  - input.{method}.params → 引数バリデーションテスト
  - input.{method}.returns → 戻り値の型・構造テスト
  - rules[] → 各 R-ID の正常/異常テスト

  subtype: service:
  - state.lifecycle → 状態遷移テスト（開始→実行中→停止等）
  - side_effects → 副作用の発生/不発生テスト

  subtype: repository:
  - CRUD roundtrip → save → find で同じデータが返るテスト
  - 存在しないキー → null 返却テスト
  - 削除後 → find で null テスト
  - 上書き → 最新データが返るテスト
  - storage 固有テスト（アトミック書き込み等、rules から導出）

出力: tests/contracts/level2/CON-{name}.test.ts (or 対応拡張子)
```

**テストデータビルダー規約（制約干渉対策）**:

テスト対象の制約を壊すとき、他の全制約を満たすデータを使用する。

```
原則: 「対象制約だけを壊す」
1. ベースとなる valid payload/data を定義
2. テスト対象のフィールドだけを変更
3. 変更値が他の制約に干渉しないことを確認
4. コメントで干渉回避の理由を明記
```

例: quantity の max 境界テストで stock 不足テストと干渉する場合
→ `quantity: 10`（max: 99 以内）+ 低在庫商品を用意

**トレーサビリティコメント**:

各テストファイルの先頭に Contract 参照を埋め込む。

```
// @generated by /test-from-contract
// @contract CON-order-create
// @implements FR-001 (from contract links.implements)
```

各 describe/it ブロックにも導出元を明記:

```
// Derived from: business_rules[BR-001]
// Derived from: input.body.items (required: true)
```

```

```

## 出力ディレクトリ構造

```
tests/contracts/
├── level1/
│   └── CON-{name}.test.ts       # 構造検証（即 GREEN）
├── level2/
│   └── CON-{name}.test.ts       # 実装検証（RED スタブ）
└── helpers/
    ├── contract-loader.ts        # YAML パースユーティリティ（共通）
    └── {type}-helpers.ts         # タイプ別テストヘルパー（スタブ）
```

## 原則

| 原則 | 説明 |
|------|------|
| Contract が唯一の真実源 | テストは Contract フィールドから導出。Contract にない仕様はテストしない |
| RED は可視化 | Level 2 は `it.skip` でなく実際に FAIL する形式で生成 |
| 対象制約だけを壊す | テストデータは対象以外の全制約を満たす値を使用 |
| トレーサビリティ埋め込み | `@contract` + `@implements` コメントで追跡可能にする |
| 冪等生成 | `// @generated` マーカー付きファイルは再実行で上書き可能 |
| フレームワーク適応 | 検出したフレームワークに合わせて拡張子・アサーション API を切り替え |

## エラーハンドリング

| エラー | 対応 |
|--------|------|
| `.blueprint/` が存在しない | `/spec` の実行を案内 |
| Contract が 0 件 | `/spec` で Contract を作成するよう案内 |
| Contract YAML パースエラー | エラー内容を報告、該当 Contract をスキップして続行 |
| Contract schema 不正（必須フィールド欠落） | 警告を表示、可能な範囲で Level 1 テストを生成 |
| テストフレームワーク未検出 | ユーザーに確認、デフォルト Vitest を提案 |
| 未対応フレームワーク指定 | Vitest 形式で生成 + 手動調整が必要な旨を警告 |
| 既存テストファイルあり（`@generated` マーカー付き） | 上書き（自動生成ファイル） |
| 既存テストファイルあり（`@generated` マーカーなし） | 上書き前にユーザー確認 |
| 部分生成失敗（N 件中 M 件成功） | 成功分は保持、失敗分を報告して続行 |
| deprecated Contract | 警告表示、デフォルトではスキップ（ユーザー強制可） |
