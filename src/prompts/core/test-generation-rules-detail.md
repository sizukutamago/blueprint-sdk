# テスト生成ルール集

Contract YAML からテストコードを生成する際のルール集。
AI はこのドキュメントを参照してテストを生成する。

> **参照**: `core/contract-schema.md` のテスト導出パターン表、`core/test-from-contract.md` のワークフロー

## Level 1: 構造検証テスト

Level 1 テストは Contract YAML の構造を検証する。実装コード不要で即 GREEN になる。

### 共通チェック項目（全タイプ）

| チェック項目 | 検証内容 |
|------------|---------|
| YAML パース | ファイルが有効な YAML であること |
| id | `CON-*` プレフィックスであること |
| type | `api` / `external` / `file` / `internal` のいずれか |
| subtype | type が `internal` の場合: `service` / `repository` |
| version | SemVer 形式 (`X.Y.Z`) |
| status | `draft` / `active` / `deprecated` のいずれか |
| owner | 空でないこと |
| updated_at | 日付形式であること |
| links | オブジェクトであり、各フィールドが配列 |

### タイプ別チェック項目

#### api 型

| フィールド | 検証内容 |
|-----------|---------|
| method | HTTP メソッド文字列 |
| path | `/` で始まるパス |
| input | オブジェクト（body が存在） |
| output | success と errors が存在 |
| business_rules | 配列、各要素に id と rule |
| state_transition | 存在する場合: entity, initial, transitions |

#### external 型

| フィールド | 検証内容 |
|-----------|---------|
| provider | 空でない文字列 |
| endpoint | method と url が存在 |
| request | body が存在 |
| response | success と errors が存在 |
| constraints | 配列、各要素に id と constraint |

#### file 型

| フィールド | 検証内容 |
|-----------|---------|
| direction | `import` / `export` のいずれか |
| format | `csv` / `tsv` / `json` 等 |
| columns | 配列、各要素に name と required |
| processing_rules | 配列、各要素に id と rule |
| result | success と error の構造定義 |

#### internal 型

| フィールド | 検証内容 |
|-----------|---------|
| subtype | `service` / `repository` のいずれか |
| description | 空でない文字列 |
| input | オブジェクト（メソッド定義が 1 つ以上） |
| rules | 配列、各要素に id と description |
| storage (repository) | オブジェクト（type が存在） |
| state (service) | 存在する場合: manages が空でない |

### Level 1 テスト構造例（Vitest）

```typescript
// @generated by /test-from-contract
// @contract CON-order-create

describe("CON-order-create: Contract Structure (Level 1)", () => {
  const contract = loadContract("api/order-create.contract.yaml");

  describe("メタデータ", () => {
    it("id が CON- プレフィックスであること", () => {
      expect(contract.id).toMatch(/^CON-/);
    });

    it("必須メタデータが全て存在すること", () => {
      expect(contract.type).toBeDefined();
      expect(contract.version).toMatch(/^\d+\.\d+\.\d+$/);
      expect(contract.status).toMatch(/^(draft|active|deprecated)$/);
      expect(contract.owner).toBeTruthy();
      expect(contract.updated_at).toBeTruthy();
    });
  });

  describe("タイプ固有フィールド", () => {
    it("api 型の必須フィールドが存在すること", () => {
      expect(contract.method).toBeDefined();
      expect(contract.path).toMatch(/^\//);
      expect(contract.input).toBeDefined();
      expect(contract.output).toBeDefined();
      expect(contract.business_rules).toBeInstanceOf(Array);
    });
  });

  describe("リンク", () => {
    it("links 構造が正しいこと", () => {
      expect(contract.links).toBeDefined();
      if (contract.links.implements) {
        expect(contract.links.implements).toBeInstanceOf(Array);
      }
    });
  });
});
```

## Level 2: 実装検証テスト

Level 2 テストは Contract の仕様に基づいて実装を検証する。
生成時点では RED スタブ（FAIL する形式）で出力する。

### RED スタブの AAA 骨格（必須）

全ての Level 2 テストに **Arrange / Act & Assert** の骨格コメントを含めること。

```typescript
it("テスト名", () => {
  // Derived from: {contract_field}
  // @contract {CON-ID}

  // Arrange
  const payload = validPayload();    // valid base からテスト対象だけ変更
  payload.targetField = invalidValue; // 単一変更原則

  // Act & Assert
  // TODO: call API/function with payload, expect {expected_status} {expected_error}
  expect.fail("RED: implement to verify {what}");
});
```

**ルール**:
- `validPayload()` は関数として定義（テスト間のミューテーション干渉を回避）
- Arrange で valid base + 単一変更を明記
- Act & Assert の TODO に期待するステータスコード/エラーコードを記載
- GREEN 化時に `expect.fail` 行を削除して実際のアサーションに置き換える

### フィールド別導出手順

> **Note**: 以下のパターン例は導出ロジックの説明に焦点を当てており、一部は省略形で記載している。
> **実際のテスト生成時は、上記の「RED スタブの AAA 骨格」に必ず従うこと。**
> `validPayload()` + Arrange / Act & Assert コメントは全テストに必須。

#### `required: true`

Contract の入力フィールドに `required: true` がある場合:

- **テスト**: 対象フィールドを空/欠落にして適切なエラーが返ることを検証
- **テスト名**: `"{field} が未送信の場合はエラー"`
- **注意**: 必須フィールドが複数ある場合、他の必須フィールドは valid な値を設定

```typescript
it("items が未送信の場合はエラー", () => {
  // Derived from: input.body.items (required: true)
  // @contract CON-order-create

  // Arrange
  const payload = { ...validPayload() };
  delete payload.items;

  // Act & Assert
  // TODO: call API/function with payload, expect 400 error with "items is required"
  expect.fail("RED: implement to verify items is required");
});
```

#### `required: true` + `default` の矛盾検出

Contract で同一フィールドに `required: true` と `default` が両方指定されている場合、**論理矛盾**（P1）として報告する。

| required | default | 意味 | テスト生成 |
|----------|---------|------|-----------|
| `true` | なし | 必須フィールド。欠落はエラー | 欠落エラーテストを生成 |
| `true` | あり | **矛盾**: 必須なのにデフォルトは不要 | P1 finding を生成。default 適用テストは**出さない** |
| `false` | あり | 任意フィールド。省略時は default 適用 | 省略→default 適用テストを生成 |
| `false` | なし | 任意フィールド。省略時は null/未設定 | 省略 OK テストを生成 |

**矛盾検出時のアクション**:
1. テスト生成を続行する（ブロックしない）
2. テストファイルの先頭に `// WARNING: Contract矛盾 - {field} は required:true かつ default あり` を出力
3. Review Gate の Test Reviewer に P1 finding として報告

#### `min: N` / `max: N`

数値/文字列長/配列長の境界を検証する:

- **テスト 3 ケース**:
  1. `N-1`（min なら NG、max なら OK）
  2. `N`（境界値、OK）
  3. `N+1`（min なら OK、max なら NG）
- **テスト名**: `"{field}: {N+1} は拒否（max: {N}）"` / `"{field}: {N} は OK（境界値 max）"`

```typescript
it("items[].quantity: 100 は拒否（max: 99）", () => {
  // Derived from: input.body.items[].quantity (max: 99)
  // Note: other fields use valid values to avoid constraint collision

  // Arrange
  const payload = validPayload();
  payload.items[0].quantity = 100; // max + 1

  // Act & Assert
  // TODO: call API/function with payload, expect 400 error
  expect.fail("RED: implement to verify quantity max boundary");
});

it("items[].quantity: 99 は OK（境界値 max）", () => {
  // Derived from: input.body.items[].quantity (max: 99)

  // Arrange
  const payload = validPayload();
  payload.items[0].quantity = 99; // exactly max

  // Act & Assert
  // TODO: call API/function with payload, expect 2xx success
  expect.fail("RED: implement to verify quantity at max boundary passes");
});
```

#### `min_items: N`

- **テスト 2 ケース**:
  1. `N-1` 要素（最小数未満 → NG）
  2. `N` 要素（境界値 → OK）
- **テスト名**: `"{field}: {N-1} 件は拒否（min_items: {N}）"`

```typescript

```

- **テスト 2 ケース**:
  1. `N` 文字（境界値 → OK）
  2. `N+1` 文字（超過 → NG）
- **テスト名**: `"{field}: {N+1} 文字は拒否（max_length: {N}）"`


#### `pattern`

正規表現パターンに対するマッチ/不一致:

- **テスト 2 ケース**:
  1. パターンに合致する値 → OK
  2. パターンに合致しない値 → エラー
- **テスト名**: `"{field}: パターンに合致しない値は拒否"`

#### `enum: [a, b, c]`

列挙値の検証:

- **テスト**: 無効値テスト（enum にない値 → エラー）
- **テスト名（無効値）**: `"{field}: 無効な値 'x' は拒否"`
- **オプション**: 全有効値テスト（各 enum 値で正常動作確認）

```typescript
it("category: 無効な値 'unknown' は拒否", () => {
  // Derived from: columns[category].enum: [electronics, clothing, food, books, other]

  // Arrange
  const payload = validPayload();
  payload.category = "unknown";  // enum にない値

  // Act & Assert
  // TODO: call API/function with payload, expect 400 / validation error
  expect.fail("RED: implement to verify invalid enum value is rejected");
});
```

#### `default: X`

デフォルト値の適用:

- **テスト**: フィールドを省略して、デフォルト値 X が適用されることを検証
- **テスト名**: `"{field}: 省略時はデフォルト値 '{X}' が適用される"`

### タイプ固有の導出手順

#### api 型: `business_rules[]`

各ビジネスルールの正常系 + 異常系:

- **テスト名に BR-ID を含める**: `"BR-001: サーバー側で金額再計算"`
- **正常系**: ルールが正しく適用されることを検証
- **異常系**: ルール違反時のエラーを検証（該当する場合）

```typescript

```

#### api 型: `state_transition`

- **初期状態テスト**: エンティティ作成時の状態が `initial` であること

- **拒否遷移テスト**: `transitions[from]` に含まれない状態への遷移が拒否されること
- **終端状態テスト**: 遷移先が空 (`[]`) の状態からの遷移試行

  - 金額検証: サーバー計算値を使用

#### file 型: `columns[]` の詳細検証

各カラムの制約を個別にテスト:

| カラム属性 | テスト導出 |
|-----------|----------|
| `required: true` | 空値テスト |
| `pattern` | マッチ/不一致テスト |
| `max_length: N` | N 文字 OK / N+1 文字 NG |
| `min: N` | N-1 NG / N OK |
| `enum: [...]` | 無効値テスト |

#### file 型: ファイルレベル制約

|------|----------|
| `has_header: true` | ヘッダー行なし → エラー |
| `max_rows: N` | N 行 OK / N+1 行 NG |
| `max_file_size` | サイズ超過 → エラー |
| 必須カラム欠落 | required カラムが CSV ヘッダーにない → エラー |

## 制約干渉（Constraint Collision）対策

### 問題

テスト対象の制約を検証するために無効な値を使うと、別の制約に先に引っかかることがある。

### 例

| 目的 | 試みた入力 | 干渉する制約 | 結果 |
|------|-----------|------------|------|
| 在庫不足テスト | quantity: 99999 | max: 99 | 400 invalid_input（在庫チェック前にバリデーション失敗） |
| 金額上限テスト | price: 999999 | max_length: 10 | カラムバリデーションエラー |
| 遷移拒否テスト | paid → cancelled | depends_on: payment must exist | 先に payment 依存でエラー |

### ルール

1. **ベースデータ**: 全制約を満たす valid なデータセットを定義（`validPayload`, `validCsvRow` 等）
2. **単一変更**: テスト対象のフィールド/条件だけを変更
3. **干渉回避値**: 変更値が他の制約範囲内に収まることを確認
4. **コメント明記**: `// Note: quantity=10 (within max:99) to avoid input validation`
5. **セットアップ分離**: 状態遷移テストでは前提状態のセットアップを独立させる

### テストデータビルダーパターン

```typescript
// ベース: 全制約を満たす valid payload
const validPayload = {
  items: [{ product_id: "PROD-001", quantity: 1 }], // min:1, max:99
  shipping_address: { zip: "100-0001" },             // pattern match
  payment_method_id: "pm_test_valid",                // required
};

// 単一フィールド変更でテスト
it("items が空配列の場合はエラー", () => {
  const payload = { ...validPayload, items: [] }; // min_items: 1 違反
  // ...
});
```
