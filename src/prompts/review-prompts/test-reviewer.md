# Test Review Swarm プロンプト

パイプライン Stage 2 後の Test Review Gate で使用するレビュープロンプト。
3 エージェントが並列でレビューし、findings を返す。

## 共通入力

各エージェントに以下を渡す:
- テストファイルパスリスト（`tests/contracts/level1/*.test.*`, `tests/contracts/level2/*.test.*`）
- ヘルパーファイル（`tests/contracts/helpers/`）
- 対応する Contract YAML ファイル（`.blueprint/contracts/**/*.contract.yaml`）
- `core/test-from-contract.md` の内容（テスト導出ルール参照）
- `core/review-criteria.md` の内容（P0/P1/P2 定義、Gate 判定基準、Severity ガバナンスルール）

## 共通出力フォーマット

```yaml
reviewer: "{エージェント名}"
gate: "test"
findings:
  - severity: P0 | P1 | P2
    target: "tests/contracts/level2/CON-xxx.test.ts"
    field: "describe > it block name"
    message: "問題の説明"
    suggestion: "修正提案"
summary:
  p0: 0
  p1: 0
  p2: 0
```

> **Note**: REVISE サイクルで severity を変更する場合は `core/review-criteria.md` の Severity ガバナンスルールに従い、`disposition` + `disposition_reason` + `original_severity` を記録すること。

## Agent 1: Coverage Validator

### 役割

Contract のフィールドがテストで漏れなくカバーされているかを検証する。

### チェック手順

```
1. Level 1（構造検証テスト）の網羅性:

   各 Contract ごとに:
   - Level 1 テストファイルが存在するか → 欠落: P0
   - YAML パース可能テストがあるか → 欠落: P1
   - 必須メタデータテスト（id, type, version, status, owner, updated_at）→ 欠落: P1
   - links 構造テスト → 欠落: P2
   - タイプ別必須フィールドテスト → 欠落: P1

2. Level 2（実装検証テスト）の網羅性:

   各 Contract ごとに:
   - Level 2 テストファイルが存在するか → 欠落: P0

   api Contract:
   - input の全 required: true フィールドに空値テストがあるか → 欠落: P1
   - min/max 制約のあるフィールドに境界値テスト（min-1, min, max, max+1）があるか → 欠落: P1
   - pattern 制約のあるフィールドにマッチ/不一致テストがあるか → 欠落: P1
   - enum 制約のあるフィールドに全有効値 + 無効値テストがあるか → 欠落: P1
   - business_rules の全 BR-xxx に対応する it ブロックがあるか → 欠落: P1
   - state_transition の全遷移（許可 + 拒否）にテストがあるか → 欠落: P1
   - errors の全エラーに status/code テストがあるか → 欠落: P1

   external Contract:
   - constraints の全 EC-xxx に対応する it ブロックがあるか → 欠落: P1
   - response.errors の全 type にハンドリングテストがあるか → 欠落: P1
   - 固定値フィールドの値検証テストがあるか → 欠落: P2

   file Contract:
   - columns の全 max_length にテストがあるか → 欠落: P1
   - max_rows テストがあるか（定義されている場合）→ 欠落: P1
   - processing_rules の全 PR-xxx に対応する it ブロックがあるか → 欠落: P1
   - result 構造テストがあるか → 欠落: P1

3. AAA スケルトンチェック:

   Level 2 の各 it ブロックについて:
   - Arrange / Act / Assert（または Act & Assert）のコメント骨格があるか → 欠落: P1
   - Arrange セクションに valid base + 単一フィールド変更の構造があるか → 欠落: P2
   - RED スタブ（expect.fail 等）のメッセージが「RED: implement to verify {what}」形式か → 不一致: P2

4. テスト数の妥当性チェック:
   - Level 2 テスト数が 0 → P0
   - Contract のフィールド数に対してテスト数が著しく少ない（< フィールド数 × 0.5）→ P2
```

## Agent 2: Constraint Collision Checker

### 役割

テストデータが「対象制約だけを壊す」原則に従っているかを検証する。
境界値テストが他の制約に干渉していないかを確認する。

### チェック手順

```
1. Valid base payload の存在チェック:
   - テストファイル内に valid payload / valid data の定義があるか
   → 存在しない: P1（テストデータの基準がない）
   - valid base が関数（validPayload() 等）として定義されているか
   → オブジェクトリテラル直書き: P2（ミューテーション干渉のリスク）

2. 境界値テストの干渉チェック:

   各 min/max 境界値テストについて:
   a. テスト対象フィールドを特定
   b. テストで使用している値を確認
   c. その値が他のフィールドの制約を侵していないか:
      - quantity: max+1 のテストで、stock 制約を超えていないか
      - price: min-1 のテストで、total_amount 制約に干渉していないか
      - items 配列: min_items-1 のテストで、required: true を侵していないか
   → 干渉検出: P1

3. テストデータの変更粒度チェック:
   - 各テストケースが valid base から 1 フィールドだけ変更しているか
   - 複数フィールドを同時に変更している場合:
     → テスト対象が複合条件でない場合: P1
     → テスト対象が複合条件の場合: P2（コメントで意図が明記されていれば OK）

4. 干渉回避コメントの存在チェック:
   - 境界値テストに「他の制約を侵さない値を使用」等のコメントがあるか
   → コメントなし: P2（テスト意図の明確化推奨）

5. enum テストの排他性チェック:
   - 無効値テストで使用している値が他の enum の有効値と重複していないか
   → 重複: P2
```

## Agent 3: Traceability Auditor

### 役割

テストファイルのトレーサビリティコメントが完全かを検証する。
`/generate-docs` が Contract → テスト → コード の chain を追跡できることを保証する。

### チェック手順

```
1. ファイルヘッダーコメントチェック:

   各テストファイルに以下が存在するか:
   - // @generated by /test-from-contract → 欠落: P1
   - // @contract CON-{name} → 欠落: P0（Contract との紐付けが不明）
   - // @implements FR-xxx → 欠落: P2（Contract の links.implements から導出）

   @contract の値が対応する Contract YAML の id と一致するか → 不一致: P0

2. Derived from コメントチェック:

   Level 2 の各 describe ブロック:
   - ブロック内に // Derived from: のコメントがあるか → 欠落: P1
   - コメントの参照先が Contract 内に実在するか:
     - business_rules[BR-xxx] → Contract に BR-xxx があるか
     - input.body.xxx (required: true) → Contract に該当フィールドがあるか
     - constraints[EC-xxx] → Contract に EC-xxx があるか
     - processing_rules[PR-xxx] → Contract に PR-xxx があるか
   → 参照先不在: P1

3. Level 1 カバレッジチェック:
   - .blueprint/contracts/ の全 Contract に対応する Level 1 テストファイルが存在するか
   → 欠落: P0

4. Level 2 カバレッジチェック:
   - .blueprint/contracts/ の全 active + draft Contract に対応する Level 2 テストファイルが存在するか
   → 欠落: P1

5. ヘルパーファイルの存在チェック:
   - tests/contracts/helpers/contract-loader.{ts|js|py} が存在するか → 欠落: P1
   - 各タイプの Contract に対応するタイプ別ヘルパーが存在するか → 欠落: P2

6. RED スタブ形式チェック:
   - Level 2 テストが it.skip ではなく実際に FAIL する形式（expect.fail / throw / pytest.fail / t.Fatal）か
   → it.skip 使用: P1（RED が可視化されない）

7. required + default 矛盾チェック:
   - Contract で required: true と default が同一フィールドに指定されている場合:
     → テスト側に P1 finding コメントが記載されているか → 欠落: P1
     → default 適用テストが生成されていないか → 生成されている: P1（矛盾するテスト）
   - required: false + default の場合: 省略時デフォルト適用テストが存在するか → 欠落: P1
```
