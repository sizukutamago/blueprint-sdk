# Contract Review プロンプト

Contract YAML の品質をレビューする。
3 エージェントが並列でレビューし、findings を返す。

## 重要: Severity 基準

- **P0 (Critical)**: YAML パース不可、テスト/実装ステージが完全に動作不能になる問題のみ
- **P1 (Important)**: テスト導出を妨げる曖昧さ、欠落した型情報、不整合
- **P2 (Minor)**: 命名規約、メタデータ欠落、推奨事項

> P0 は「パイプラインが壊れる」問題に限定すること。スキーマ形式やスタイルは P1/P2。

## チェック対象

`.blueprint/contracts/` 配下の全 YAML ファイルを読み込んでレビューする。

---

## Agent 1: 構造チェック

1. YAML としてパース可能か → 不可: P0
2. `id` が存在するか → 欠落: P0
3. `type` が定義されているか → 欠落: P1
4. 関数/メソッドの `returns` が定義されているか → 欠落: P1
5. `depends_on` の参照先が実在するか → 実在しない: P1
6. `constraints` が存在するか → 欠落: P2

---

## Agent 2: 完全性チェック

1. 各入力パラメータに `type` が定義されているか → 未定義: P1
2. 数値パラメータに範囲制約（min/max）があるか → 未定義: P1
3. 異常系の動作が定義されているか → 未定義: P1
4. 戻り値の型と説明が明確か → 不明確: P1
5. ID の命名形式（推奨: CON-*）→ 非準拠: P2
6. version/status/owner 等のメタデータ → 欠落: P2

---

## Agent 3: テスト可能性チェック

1. 各関数の invariants/constraints からテストケースを導出できるか
   → 曖昧で導出不可: P1
2. 入力パラメータの境界値テストが導出可能か
   → 制約なしで導出不可: P1
3. 曖昧表現（「適切に」「必要に応じて」等）がないか
   → 検出: P2
4. エラーケースの発生条件が明確か
   → 不明確: P1
